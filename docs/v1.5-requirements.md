# Jupiter v1.5 — Views & Workflow

**Status:** Planned  
**Target:** v1.5.0  
**Jira alignment:** [List view](https://www.atlassian.com/software/jira/features), calendar, workflow rules, board quick filters (Spring 2026 themes)  
**Depends on:** v1.4 (sprints, board config, custom fields, JQL-lite)

---

## Summary

v1.5 adds two new **work views** (List and Calendar) and strengthens **workflow control** with transition rules and **quick filters** on board/backlog — closing the biggest gaps vs Jira’s core navigation (board · backlog · list · calendar · timeline).

---

## Goals

1. Let teams manage issues in a **spreadsheet-style list** without leaving the project.
2. Surface **due dates** on a calendar and in issue details.
3. Enforce **who can move issues to which status** (workflow transitions).
4. Save **one-click filter chips** on board and backlog (Jira-style quick filters).

## Non-goals (v1.5)

- Custom dashboards (v1.7)
- Full JQL parity
- **Email-based auth** (sign-up, verification, reset) — **v1.6** ([spec](./v1.6-auth-requirements.md))
- Guest access / issue email notifications
- AI / automations
- Postgres migration of issue data (stay on Zustand for v1.5)

## User stories

| ID | As a… | I want to… | So that… |
|----|--------|------------|----------|
| US-1 | team member | open a List tab in my project | I can scan and edit many issues like a spreadsheet |
| US-2 | team member | sort and show/hide columns in List view | I focus on the fields I care about |
| US-3 | team member | see issues on a Calendar by due date | I know what is due this week |
| US-4 | team member | set a due date on an issue | it appears on the calendar and in list |
| US-5 | project lead | define allowed status transitions per role | viewers cannot drag items to Done |
| US-6 | team member | get clear feedback when a transition is blocked | I know why a drag or status change failed |
| US-7 | team member | apply saved quick filters on board/backlog | I switch between “My open” and “Bugs only” in one click |
| US-8 | project lead | create and name quick filters for the project | the whole team shares the same shortcuts |

## Functional requirements

### FR-1 — Due date field

| ID | Requirement |
|----|-------------|
| FR-1.1 | Add optional `dueDate` (ISO date string, `YYYY-MM-DD`) to `Issue` schema and Zustand store. |
| FR-1.2 | Show due date in issue dialog (date picker); log changes in activity. |
| FR-1.3 | Include due date in seed data for a subset of demo issues. |
| FR-1.4 | Support filter/sort by due date in List view and JQL-lite (`dueDate` field). |
| FR-1.5 | Visual indicator when due date is past and status ≠ done (overdue badge). |

### FR-2 — List view

| ID | Requirement |
|----|-------------|
| FR-2.1 | New route: `/projects/[key]/list` with tab **List** in project shell (between Backlog and Timeline). |
| FR-2.2 | Table columns (default): Key, Type, Summary, Status, Priority, Assignee, Due date, Sprint, Story points. |
| FR-2.3 | Click row or key opens issue dialog; inline edit summary on double-click (optional stretch: status dropdown). |
| FR-2.4 | Column sort (asc/desc) on Key, Status, Priority, Due date, Updated. |
| FR-2.5 | Reuse `IssueFiltersBar` + sprint filter pattern from board view. |
| FR-2.6 | Persist visible columns + sort in `localStorage` key `jupiter-list-columns:{projectId}`. |
| FR-2.7 | Group-by toggle: **Flat** \| **Epic** (indent children under parent epic). |
| FR-2.8 | Keyboard: arrow keys move row focus; Enter opens issue. |
| FR-2.9 | Empty state when no issues match filters. |

### FR-3 — Calendar view

| ID | Requirement |
|----|-------------|
| FR-3.1 | New route: `/projects/[key]/calendar` with tab **Calendar** in project shell. |
| FR-3.2 | Month view default; toggle week view (stretch goal — month-only acceptable for MVP). |
| FR-3.3 | Show issues with `dueDate` set; color by type or priority. |
| FR-3.4 | Click day cell → create issue pre-filled with that due date (if `issues.create`). |
| FR-3.5 | Click issue chip → open issue dialog. |
| FR-3.6 | Drag issue chip to another day updates `dueDate` (if `issues.edit`). |
| FR-3.7 | Issues without due date do not appear on calendar (with link: “View issues missing due dates in List”). |

### FR-4 — Workflow transition rules

| ID | Requirement |
|----|-------------|
| FR-4.1 | Add `transitionRules` on `Project` (or dedicated workflow store): map `fromStatus → toStatus[]` allowed per role group (`admin`, `lead`, `member`, `viewer`). |
| FR-4.2 | Default rules: all transitions allowed for admin/lead/member; viewer cannot transition. |
| FR-4.3 | UI in **Board config** (or new **Workflow** sub-page): matrix or from→to checklist editor. |
| FR-4.4 | Enforce rules in: Kanban drag-and-drop, backlog status change, issue dialog status select. |
| FR-4.5 | Blocked transition: revert drag, toast/message “Transition not allowed for your role”. |
| FR-4.6 | Audit log entry on blocked attempt (optional, nice-to-have). |
| FR-4.7 | Export helper: `canTransition(role, from, to, project)` in `lib/workflow.ts`. |

### FR-5 — Quick filters (board & backlog)

| ID | Requirement |
|----|-------------|
| FR-5.1 | Quick filter bar below project tabs on Board and Backlog views. |
| FR-5.2 | Built-in chips: **All**, **Assigned to me**, **Unassigned**, **Bugs**, **In progress**. |
| FR-5.3 | Project-scoped saved filters: name + `IssueFilters` payload, stored in `projects-store` or `localStorage` per project. |
| FR-5.4 | “Save current filters” action when filters differ from default. |
| FR-5.5 | Active chip visually highlighted; one primary chip active at a time (+ combinable with filter bar). |
| FR-5.6 | Lead/admin can delete custom quick filters. |

## Data model changes

```typescript
// Issue — add to issueSchema
dueDate?: string; // ISO date YYYY-MM-DD

// Project — add optional
transitionRules?: Partial<
  Record<UserRole, Partial<Record<IssueStatus, IssueStatus[]>>>
>;

// QuickFilter — new type (project-scoped)
interface QuickFilter {
  id: string;
  projectId: string;
  name: string;
  filters: IssueFilters;
  createdById: string;
}
```

**Migration:** Zustand persist rehydration; missing `dueDate` / `transitionRules` treated as undefined. Seed updates only.

- [ ] Viewer role cannot drag issue to Done when rules disallow it.

- [ ] Due date editable in issue dialog; appears in List column after save.

- [ ] No regressions on existing board, backlog, sprint board.
