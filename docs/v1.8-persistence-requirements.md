# v1.8 — Notifications, audit log & burndown persistence

**Status:** Shipped (Option A)  
**Depends on:** v1.6 auth, workspace sync (`/api/workspace`)  
**References:** [ARCHITECTURE.md](./ARCHITECTURE.md) · [DATABASE.md](./DATABASE.md)

---

## Recommendation: API first, database as source of truth

| Approach | Use for Jupiter | Why |
|----------|-----------------|-----|
| **HTTP API** (Next.js route handlers + Drizzle) | ✅ **Default for all app features** | Same pattern as `/api/auth/*` and `/api/workspace`; enforces auth, validation, and session cookies; works on Vercel serverless. |
| **Direct database** (SQL client, psql, BI tools) | ✅ **Ops & reporting only** | Migrations (`drizzle-kit push`), seeds, backups, ad-hoc analytics — not from the React client. |
| **Extend full workspace snapshot only** | 🟡 **Partial** | OK for burndown + read-state short term; becomes heavy as data grows. Prefer **targeted tables + small APIs** for v1.8. |

**Do not** expose Postgres connection strings to the browser or add a “direct DB” mode in the UI. “Direct database” in requirements means **persisted in PostgreSQL**, accessed **through the server**.

## Current state (baseline)

| Feature | What exists today | In Postgres today? |
|---------|-------------------|-------------------|
| **Notifications (feed)** | Derived from issue `activity` where user is assignee/reporter | ✅ Activity rows sync via `/api/workspace` |
| **Notifications (read/unread)** | `notifications-store` → `localStorage` only | ❌ |
| **Audit log (`/audit`)** | UI over **issue** `activity` (filter/search) | ✅ Same `activity` table |
| **Workspace-level audit** | Not modeled (no project/sprint/member events) | ❌ |
| **Burndown snapshots** | `sprints-store.burndownSnapshots` → `localStorage` | ❌ |

So: **audit content for issue changes is already in the DB** after workspace sync; what’s missing is **read state**, **sprint burndown history**, and optional **workspace-wide events**.

---

## Goals (v1.8)

1. **Cross-device notifications** — unread state survives browser/session changes.
2. **Durable audit** — issue history remains queryable at scale; optional workspace events for admin compliance.
3. **Historical burndown** — completed sprints keep point-in-time curves for reports after reload.

## Proposed schema

### 1. `notification_reads`

Tracks which activity entries a user has seen (feed body still derived from `activity`).

```sql
-- Drizzle: composite PK (user_id, activity_id)
notification_reads (
  user_id     TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  activity_id TEXT NOT NULL REFERENCES activity(id) ON DELETE CASCADE,
  read_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, activity_id)
);
CREATE INDEX notification_reads_user_idx ON notification_reads (user_id);
```

| Field | Notes |
|-------|--------|
| `activity_id` | FK to existing `activity` row (notification id = activity id today) |

**Later (v1.9+):** optional `notifications` table for explicit events (assignment, mention) if feed rules outgrow activity.

---

### 2. `workspace_events` (optional but recommended)

Issue `activity` is **per-issue**. Workspace audit needs **project/sprint/membership** events.

```sql
workspace_events (
  id          TEXT PRIMARY KEY,
  project_id  TEXT REFERENCES projects(id) ON DELETE CASCADE,
  actor_id    TEXT NOT NULL REFERENCES users(id),
  kind        TEXT NOT NULL,  -- project.created | sprint.started | member.added | ...
  message     TEXT NOT NULL,
  metadata    JSONB,          -- { sprintId, userId, ... }
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX workspace_events_project_idx ON workspace_events (project_id);
CREATE INDEX workspace_events_created_idx ON workspace_events (created_at DESC);
```

**Audit UI:** merge `workspace_events` + issue `activity` (or tabs: “Issues” / “Workspace”).

**Alternative (smaller scope):** skip this table in v1.8 and only persist **read state + burndown**; keep `/audit` as issue activity only.

---

### 3. `burndown_snapshots`

Time series per sprint (written on sprint complete and optionally daily).

```sql
burndown_snapshots (
  id               TEXT PRIMARY KEY,
  sprint_id        TEXT NOT NULL REFERENCES sprints(id) ON DELETE CASCADE,
  recorded_at      TIMESTAMPTZ NOT NULL,
  remaining_points INTEGER NOT NULL DEFAULT 0,
  scope_points     INTEGER,  -- optional total committed at sprint start
  created_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX burndown_snapshots_sprint_idx ON burndown_snapshots (sprint_id, recorded_at);
```

**Alternative:** `sprints.burndown_data JSONB` — fewer joins, harder to query across projects. Prefer **normalized table** for reports.

## API design (recommended)

Extend REST surface; keep workspace snapshot for bulk issue/project data.

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/api/notifications` | Feed for current user (wraps activity query + read flags) |
| `POST` | `/api/notifications/read` | `{ activityIds: string[] }` mark read |
| `POST` | `/api/notifications/read-all` | Mark all visible as read |
| `GET` | `/api/audit` | Paginated issue `activity` + optional `workspace_events` (admin/viewer) |
| `GET` | `/api/sprints/:id/burndown` | Snapshots for one sprint |
| `PUT` | `/api/sprints/:id/burndown` | Append snapshot (server or client after complete) |

**Auth:** all routes require `getCurrentUser()`; audit requires `audit.view`; writes respect existing role rules.

**Sync strategy (client):**

```mermaid
flowchart LR
  subgraph keep [Keep as-is]
    WS[WorkspaceSync snapshot]
  end
  subgraph new [v1.8 targeted APIs]
    N[/api/notifications]
    A[/api/audit]
    B[/api/sprints/.../burndown]
  end
  Z[Zustand] --> WS
  Z --> N
  Z --> B
  AuditPage --> A
```

- **Do not** add burndown/read-state to every `PUT /api/workspace` unless implementing v1.8 as a **minimal** step (see below).

## Implementation options

### Option A — Targeted API + tables (recommended)

- Add 2–3 tables above.
- Small route handlers; hydrate notifications read state on login.
- Append burndown on `completeSprint` via API.
- **Pros:** Clear boundaries, smaller payloads, easier pagination for audit.
- **Cons:** More routes than one snapshot.

### Option B — Extend workspace snapshot only (fastest)

- Add `notificationReads`, `burndownSnapshots`, optional `workspaceEvents` to `WorkspaceSnapshot` + `saveWorkspace`.
- **Pros:** Reuses `WorkspaceSync` debounce; one migration path.
- **Cons:** Large PUTs; audit pagination awkward; couples unrelated concerns.

### Option C — Direct DB / SQL only (not recommended for product)

- Document SQL views for operators; no client change.
- **Pros:** Quick internal reporting.
- **Cons:** No cross-device UX; bypasses auth — **not a substitute for Option A or B**.

**Ship order:** **A** for production; **B** only if you need a 1–2 day bridge before splitting APIs.

## Functional requirements

### Notifications (FR-N)

| ID | Requirement |
|----|-------------|
| FR-N1 | When Postgres is configured, read/unread state is stored per user in `notification_reads`. |
| FR-N2 | `GET /api/notifications` returns the same feed rules as `useNotifications` today (assignee/reporter, exclude self, limit 30). |
| FR-N3 | Mark read / mark all read update Postgres and local store. |
| FR-N4 | Without Postgres, behavior unchanged (`localStorage` only). |
| FR-N5 | (Stretch) `@mention` creates activity kind `mention` and always notifies mentioned user. |

### Audit log (FR-A)

| ID | Requirement |
|----|-------------|
| FR-A1 | `/audit` loads issue events from Postgres when available (query `activity` with filters, not full client array). |
| FR-A2 | Pagination: `?cursor=&limit=50` on `GET /api/audit`. |
| FR-A3 | (Optional) `workspace_events` for project/sprint/member changes with same permission `audit.view`. |
| FR-A4 | Issue-level activity continues to be written on issue mutations (existing store → workspace sync or future issue API). |

### Burndown (FR-B)

| ID | Requirement |
|----|-------------|
| FR-B1 | On sprint complete, append a `burndown_snapshots` row (remaining story points). |
| FR-B2 | Reports read snapshots from API/DB, not only `localStorage`. |
| FR-B3 | Deleting a sprint deletes its snapshots (FK cascade). |
| FR-B4 | (Stretch) Daily snapshot job for active sprints. |

## Non-goals (v1.8)

- Email/push notification delivery (see v1.9 “Email notification hooks”).
- Immutable compliance archive (WORM storage, legal hold).
- Real-time WebSocket feed (polling or SWR refresh is enough).

## Migration & rollout

1. `drizzle-kit push` — new tables.
2. Backfill: empty `notification_reads`; burndown empty until sprints complete again.
3. Deploy API routes; feature-flag client to call APIs when `GET /api/auth/config` adds `workspaceApi: true` or reuse existing DB detection.
4. Update [DATABASE.md](./DATABASE.md) and [ARCHITECTURE.md](./ARCHITECTURE.md).

## Acceptance criteria

- [ ] Two browsers, same user: mark notification read in A, refresh B → still read.
- [ ] Complete sprint on Vercel → burndown report shows snapshot after hard refresh.
- [ ] `/audit` returns events with Postgres-only seed (no localStorage dependency).
- [ ] `npm test` + `npm run build` pass.
