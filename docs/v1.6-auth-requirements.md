# Jupiter v1.6 ŌĆö Authentication (email-based)

**Status:** Planned  
**Target:** v1.6.0  
**Jira alignment:** [Atlassian account](https://www.atlassian.com/software/jira) ŌĆö users sign in with **email** (not a separate username login)  
**Builds on:** v1.4 auth API, Drizzle `users` + `sessions`, OrbStack/Docker Postgres deploy

---

## Summary

Replace demo **username** login with **standard email authorization**: users authenticate with **email + password**, verify ownership of the email address, and recover access via password reset. Sessions remain **HTTP-only cookies** on the server (no tokens in `localStorage`).

**Today (v1.4ŌĆōv1.5):** optional Postgres sessions; login accepts **username**; demo accounts in seed; client fallback when `DATABASE_URL` is unset.

**Target (v1.6):** email is the **unique login identifier**; username becomes an optional display handle; full flows for sign-up, verification, and reset.

## Goals

1. Match common SaaS / Jira expectations: **ŌĆ£Sign in with your work emailŌĆØ**.
2. Keep server-side sessions (existing `/api/auth/*` pattern).
3. Work in **Docker/OrbStack** (`DATABASE_URL` required for auth in production).
4. Preserve **local-only demo mode** for `npm run dev` without Postgres (development only).

## Functional requirements

### FR-A1 ŌĆö Identity model

| ID | Requirement |
|----|-------------|
| FR-A1.1 | `users.email` is **unique**, normalized (trim, lowercase) before store and lookup. |
| FR-A1.2 | **Login identifier** is `email` only (not `username`). |
| FR-A1.3 | `username` is optional display handle; auto-generated from email local-part if omitted at sign-up. |
| FR-A1.4 | Passwords stored as **scrypt** hash (`salt:hash`) ŌĆö existing `hashPassword` / `verifyPassword`. |
| FR-A1.5 | Password policy: min 8 chars, max 128; zod validation on all auth bodies. |
| FR-A1.6 | Roles unchanged: `admin` \| `lead` \| `member` \| `viewer`. |

|----|-------------|

| FR-A2.3 | On failure: `401` with generic message (ŌĆ£Invalid email or passwordŌĆØ) ŌĆö no user enumeration. |
| FR-A2.4 | Login UI: email field (`type="email"`, `autoComplete="email"`), password field. |

## Security requirements

- Validate all inputs with **Zod**; sanitize display names (length limits).
- **Rate limiting** on login, register, forgot-password, resend.
- Constant-time password verify (existing scrypt helper).
- Reset/verify tokens: cryptographically random, hashed at rest if stored in DB.
- **CORS** not applicable to same-origin App Router routes.
- Align with workspace **security.mdc**: no secrets in repo; use env vars.
