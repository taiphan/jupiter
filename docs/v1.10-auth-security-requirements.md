# Jupiter v1.10 ‚Äî Gmail outbound mail & TOTP two-factor authentication

**Status:** Shipped (`1.10.0`)  
**Target:** v1.10.0  
**Builds on:** [v1.6 email auth](./v1.6-auth-requirements.md) (register, verify, reset), [v1.7 Google Sign-In](./v1.7-google-sign-in-requirements.md)  
**Architecture:** [ARCHITECTURE.md](./ARCHITECTURE.md)

---

## Summary

Two security upgrades for production accounts:

1. **Real verification (and reset) email** ‚Äî after registration, Jupiter sends a verification message to the user‚Äôs inbox using **Google Gmail SMTP** (not console logs).
2. **TOTP 2FA** ‚Äî users can enroll an authenticator app (QR scan) and must enter a 6-digit code at sign-in for stronger account protection.

> **Terminology:** Gmail uses **SMTP to send** mail (`smtp.gmail.com`). **IMAP** (`imap.gmail.com`) is for *reading* an inbox. Jupiter v1.10 uses **SMTP for outbound** transactional email. Optional future work could use IMAP to monitor a bounce mailbox ‚Äî that is **out of scope** for v1.10.

---

## Goals

| # | Goal |
|---|------|
| G1 | New users receive a **delivered** verification email within minutes of sign-up (production). |
| G2 | Password-reset emails use the same trusted mail pipeline. |
| G3 | Users can enable **TOTP 2FA** (Google Authenticator, Authy, 1Password, etc.). |
| G4 | Sign-in with 2FA enabled requires **something you know** (password) + **something you have** (TOTP). |
| G5 | Secrets and credentials stay in **environment variables** ‚Äî never in git. |

## Non-goals (v1.10)

- SMS / voice OTP
- WebAuthn / passkeys (FIDO2) ‚Äî future v2.x
- Hardware security keys (U2F) ‚Äî future
- **Reading** mail via IMAP (inbox sync, bounce processing)
- Forced 2FA for entire workspace (org policy) ‚Äî stretch v2.x
- Email 2FA (magic link as second factor) ‚Äî use TOTP only
- Google Sign-In users required to also set password + 2FA (see ¬ß Google OAuth accounts)

## Part A ‚Äî Gmail SMTP transactional email

### User stories

| ID | As a‚Ä¶ | I want to‚Ä¶ | So that‚Ä¶ |
|----|--------|------------|----------|
| US-M1 | new user | get a verification email in my real inbox | I can confirm I own the address |
| US-M2 | user | receive password-reset email via the same sender | I trust the message isn‚Äôt spoofed |
| US-M3 | operator | configure Gmail with env vars only | I don‚Äôt change code per environment |
| US-M4 | developer | still use console mailer locally | I don‚Äôt need Gmail credentials on every laptop |

### Functional requirements ‚Äî mail

| ID | Requirement |
|----|-------------|
| FR-M1 | `EMAIL_PROVIDER=gmail` (alias of `smtp` with Gmail defaults) sends mail via **SMTP**. |
| FR-M2 | Support **Gmail App Password** flow: `SMTP_USER` = full Gmail address, `SMTP_PASS` = 16-char app password, host `smtp.gmail.com`, port **587** (STARTTLS) default; **465** (TLS) optional via env. |
| FR-M3 | `EMAIL_FROM` must match the authenticated Gmail/Workspace sender (e.g. `Jupiter <noreply@yourdomain.com>` or the Gmail address). |
| FR-M4 | On `POST /api/auth/register`, queue/send **verify email** with link `{APP_URL}/api/auth/verify?token=‚Ä¶` (existing token rules: single-use, 24h). |
| FR-M5 | On `POST /api/auth/forgot-password`, send **reset email** with link `{APP_URL}/reset-password?token=‚Ä¶` (1h token). |
| FR-M6 | On `POST /api/auth/resend-verification`, resend verify email (rate limit unchanged: 3/hour per email). |
| FR-M7 | HTML + plain-text multipart templates; subject lines: ‚ÄúVerify your Jupiter account‚Äù, ‚ÄúReset your Jupiter password‚Äù. |
| FR-M8 | SMTP failures return **503** to client with generic message; log error server-side **without** credentials or tokens. |
| FR-M9 | `EMAIL_PROVIDER=console` remains default for local dev ‚Äî links printed to server log. |
| FR-M10 | Implement real SMTP send in `mailer.ts` (replace ‚Äúnot wired yet‚Äù stub) using `nodemailer` or native TLS socket. |

### Gmail / Google Workspace setup (operator)

1. Google Account ‚Üí **Security** ‚Üí enable **2-Step Verification** (required for App Passwords on consumer Gmail).
2. **App passwords** ‚Üí generate app password for ‚ÄúMail‚Äù / ‚ÄúOther (Jupiter)‚Äù.
3. Set env (see `env.example`):

```bash
EMAIL_PROVIDER=gmail
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_SECURE=false          # true when using port 465
SMTP_USER=you@gmail.com
SMTP_PASS=xxxx xxxx xxxx xxxx   # app password, spaces stripped in code
EMAIL_FROM=Jupiter <you@gmail.com>
```

4. **Google Workspace:** prefer domain alias + SMTP relay or dedicated `noreply@company.com` with admin-approved sender; document SPF/DKIM in Workspace admin (operator responsibility).

### Security ‚Äî mail

| ID | Requirement |
|----|-------------|
| SEC-M1 | Never commit `SMTP_PASS`, App Passwords, or `.env`. |
| SEC-M2 | TLS required in production (`SMTP_SECURE` or STARTTLS on 587). |
| SEC-M3 | Do not include password-reset or verify **tokens** in email subject lines. |
| SEC-M4 | Links use HTTPS in production (`APP_URL` / `VERCEL_PROJECT_PRODUCTION_URL`). |
| SEC-M5 | Rate limits on register / forgot / resend unchanged (anti-abuse). |

### Acceptance ‚Äî mail

- [ ] Register with real address ‚Üí email arrives in inbox (not spam folder best-effort).
- [ ] Click verify link ‚Üí `emailVerifiedAt` set ‚Üí login succeeds.
- [ ] Forgot password ‚Üí reset email received ‚Üí password change works.
- [ ] `EMAIL_PROVIDER=console` still works without SMTP env.

## Part B ‚Äî TOTP two-factor authentication (authenticator scan)

### User stories

| ID | As a‚Ä¶ | I want to‚Ä¶ | So that‚Ä¶ |
|----|--------|------------|----------|
| US-T1 | security-conscious user | enable 2FA from Settings | my account is harder to hijack |
| US-T2 | user | scan a QR code with Google Authenticator | I don‚Äôt type the secret manually |
| US-T3 | user | enter a 6-digit code after password at login | only I can complete sign-in |
| US-T4 | user | save backup codes when enabling 2FA | I can recover if I lose my phone |
| US-T5 | user | disable 2FA with password + current TOTP | I can rotate devices safely |

### Data model

| Table / column | Type | Notes |
|----------------|------|-------|
| `users.totp_secret` | text, nullable | Encrypted at rest (AES-256-GCM with `AUTH_SECRET` derived key); null = 2FA off |
| `users.totp_enabled_at` | timestamptz, nullable | Set when enrollment completes |
| `totp_backup_codes` | table | `user_id`, `code_hash`, `used_at` ‚Äî 10 codes at enroll, single-use |
| `sessions.pending_totp` | optional flag or `auth_pending_2fa` cookie | Short-lived (5 min) after password OK, before TOTP |

**Alternative:** store secret in `user_security` 1:1 table ‚Äî acceptable if cleaner.

### Functional requirements ‚Äî 2FA enrollment

| ID | Requirement |
|----|-------------|
| FR-T1 | `POST /api/auth/2fa/setup` (authenticated, verified email) returns `{ otpauthUrl, secretMasked, qrDataUrl }` ‚Äî generates pending secret **not yet active**. |
| FR-T2 | QR encodes `otpauth://totp/Jupiter:{email}?secret=BASE32&issuer=Jupiter&algorithm=SHA1&digits=6&period=30` (RFC 6238 / Google Authenticator compatible). |
| FR-T3 | `POST /api/auth/2fa/enable` body `{ code }` ‚Äî verifies TOTP against pending secret; on success sets `totp_enabled_at`, stores encrypted secret, returns **10 backup codes** (plaintext once). |
| FR-T4 | Settings UI: ‚ÄúTwo-factor authentication‚Äù card with Enable / Disable states. |
| FR-T5 | Show manual entry key (base32) for users who cannot scan QR. |

### Functional requirements ‚Äî 2FA at login

| ID | Requirement |
|----|-------------|
| FR-T6 | If user has `totp_enabled_at` set: after valid password on `POST /api/auth/login`, **do not** issue full session yet ‚Äî return `{ requires2fa: true, challengeToken }` (signed, 5 min, httpOnly cookie optional). |
| FR-T7 | `POST /api/auth/2fa/challenge` body `{ code }` or `{ backupCode }` ‚Äî validates TOTP (window ¬±1 step) or unused backup code; then create normal `jupiter_session`. |
| FR-T8 | UI: `/login/2fa` step (or modal) with 6-digit input, ‚ÄúUse backup code‚Äù link. |
| FR-T9 | Failed TOTP: max **5** attempts per challenge per 15 min ‚Üí lock challenge, force re-login. |
| FR-T10 | Google-only accounts (`passwordHash` null): 2FA optional ‚Äî if enabled, challenge after OAuth callback before session (v1.10 stretch; **password users priority**). |

### Functional requirements ‚Äî disable & recovery

| ID | Requirement |
|----|-------------|
| FR-T11 | `POST /api/auth/2fa/disable` requires `{ password, code }` (TOTP or backup code). |
| FR-T12 | `POST /api/auth/2fa/backup-codes/regenerate` requires password + TOTP; invalidates old codes. |
| FR-T13 | Admin **cannot** disable user 2FA without audit event (future workspace_events kind `security.2fa_disabled_by_admin` ‚Äî stretch). |
| FR-T14 | Losing TOTP + all backup codes: **account recovery** = admin manual reset in DB or support process (document runbook; no self-serve in v1.10). |

### Security ‚Äî 2FA

| ID | Requirement |
|----|-------------|
| SEC-T1 | TOTP secret never sent to client after enable (except during setup before confirm). |
| SEC-T2 | Backup codes stored as **scrypt hash** (like passwords); show plaintext only once at generation. |
| SEC-T3 | Use standard TOTP (HMAC-SHA1, 6 digits, 30s) ‚Äî `otplib` or equivalent vetted library. |
| SEC-T4 | Encrypt `totp_secret` at rest; key from `AUTH_SECRET`. |
| FR-T5 | 2FA challenge token single-use, bound to user id + expiry. |
| SEC-T6 | Enabling 2FA requires verified email (`emailVerifiedAt` set). |

### UI requirements

| Screen | Route / location | Notes |
|--------|------------------|-------|
| 2FA setup | `/settings` ‚Üí Security | QR + manual key + confirm code |
| Backup codes | modal after enable | Download/copy once |
| Login 2FA step | `/login` (step 2) | 6-digit OTP input |
| Disable 2FA | `/settings` | Password + TOTP confirm |

### API summary (v1.10 additions)

| Method | Path | Purpose |
|--------|------|---------|
| POST | `/api/auth/2fa/setup` | Start enrollment (QR) |
| POST | `/api/auth/2fa/enable` | Confirm TOTP + get backup codes |
| POST | `/api/auth/2fa/challenge` | Complete login after password |
| POST | `/api/auth/2fa/disable` | Turn off 2FA |
| POST | `/api/auth/2fa/backup-codes/regenerate` | New backup codes |

Login response change (breaking for API clients):

```json
// password OK, 2FA required
{ "requires2fa": true }

// password OK, no 2FA
{ "user": { ... } }
```

### Acceptance ‚Äî 2FA

- [ ] Enable 2FA ‚Üí scan QR in Google Authenticator ‚Üí enter code ‚Üí backup codes shown.
- [ ] Log out ‚Üí login with password ‚Üí prompted for TOTP ‚Üí success.
- [ ] Wrong TOTP 5 times ‚Üí challenge invalidated.
- [ ] Backup code works once, then cannot reuse.
- [ ] Disable 2FA with password + TOTP ‚Üí login without second step.
- [ ] User with 2FA: stolen password alone cannot obtain session.
